## üë§ For Human Contributors

### Understanding AI Governance
All AI tools working on this repository follow centralized instructions in `/.ai/`:

```
/.ai/
‚îú‚îÄ‚îÄ README.md              # Start here (overview of governance system)
‚îú‚îÄ‚îÄ SYSTEM.md              # Master authority (all tools inherit this)
‚îú‚îÄ‚îÄ STYLE.md               # Brand voice & 48 Laws persuasion
‚îú‚îÄ‚îÄ DOMAIN.md              # TCNA standards & NJ HIC compliance
‚îú‚îÄ‚îÄ COMPLIANCE.md          # Legal boundaries & ethics
‚îú‚îÄ‚îÄ OUTPUT_RULES.md        # Code quality standards
‚îú‚îÄ‚îÄ COPILOT.md             # GitHub Copilot adapter
‚îú‚îÄ‚îÄ GPT.md                 # ChatGPT adapter
‚îî‚îÄ‚îÄ CODEX.md               # Autonomous agent adapter
```

**Purpose**: Ensures consistent, high-quality AI-generated code and content that:
- Follows TCNA 2024 standards (technical accuracy)
- Matches brand voice ("competent professional who refuses to suffer fools")
- Complies with NJ HIC regulations (legal safety)
- Meets code quality standards (accessibility, performance, semantic HTML)

### When Working With AI Tools

**Best practices**:
1. **Trust the governance**: AI tools reference these files automatically
2. **Verify outputs**: AI should cite sources (e.g., "Per DOMAIN.md ¬ß2.2...")
3. **Report issues**: If AI violates rules, update the relevant `/.ai/` file
4. **Don't duplicate**: Add new rules to `/.ai/` files, not scattered in comments

**When AI seems wrong**:
- Ask: "Which instruction file are you following?"
- Check: Does `/.ai/SYSTEM.md` or domain files support this?
- Correct: Update the `/.ai/` file if the rule needs clarification

### Updating Instructions

**To modify AI behavior**:
1. Edit the appropriate `/.ai/` file (STYLE, DOMAIN, COMPLIANCE, OUTPUT_RULES)
2. Commit with clear message: `docs(ai): update [rule description]`
3. Changes apply immediately to all AI tools using the governance structure

**Inheritance hierarchy**:
- `SYSTEM.md` = Master (overrides everything)
- `COMPLIANCE.md` = Legal (overrides preferences)
- `DOMAIN.md`, `STYLE.md`, `OUTPUT_RULES.md` = Equal (different domains)
- Tool adapters = Thin wrappers (inherit from above)

## üöÄ Quick Start Scenarios

### Scenario 1: Generating Marketing Copy
1. AI reads **[`/.ai/STYLE.md`](.ai/STYLE.md)** for voice guidelines
2. Applies 48 Laws framework (e.g., Law 6 - bold headlines, Law 33 - pain points)
3. Uses contrast patterns ("proper method" vs. "shortcut") without naming competitors
4. Verifies technical accuracy against **[`/.ai/DOMAIN.md`](.ai/DOMAIN.md)**
5. Checks legal compliance against **[`/.ai/COMPLIANCE.md`](.ai/COMPLIANCE.md)**

### Scenario 2: Writing Code
1. AI reads **[`/.ai/OUTPUT_RULES.md`](.ai/OUTPUT_RULES.md)** for standards
2. Uses semantic HTML (`<nav>`, `<article>`, not `<div>`)
3. Uses design tokens (no hard-coded colors: `var(--color-primary)`)
4. Adds accessibility attributes (ARIA labels, alt text)
5. Follows file naming conventions (`kebab-case.html`, `camelCase.js`)
6. Runs linters (HTMLHint, ESLint) before suggesting commit

### Scenario 3: Technical Writing (Blog Post)
1. AI reads **[`/.ai/DOMAIN.md`](.ai/DOMAIN.md)** for TCNA standards
2. Uses precise terminology (e.g., "C2-rated modified thinset", not "strong glue")
3. Cites ANSI standards (e.g., "Per ANSI A108.02, LFT requires 95% coverage...")
4. Applies voice from **[`/.ai/STYLE.md`](.ai/STYLE.md)** (technical but accessible)
5. Includes disclaimers from **[`/.ai/COMPLIANCE.md`](.ai/COMPLIANCE.md)** if needed

### Scenario 4: Contract/Legal Content
1. AI reads **[`/.ai/COMPLIANCE.md`](.ai/COMPLIANCE.md)** FIRST
2. Verifies NJ HIC requirements (license #13VH10808800, 3-day rescission)
3. Checks payment restrictions (10% OR $1,000 max deposit, whichever LESS)
4. Includes all required disclosures
5. States limitations: "This is not legal advice. Consult an attorney."

## üîç Troubleshooting

### AI Generated Wrong Output
1. **Check which file it referenced**: Ask "What instruction file are you following?"
2. **Verify the rule**: Read the cited section in `/.ai/`
3. **Update if needed**: Edit `/.ai/` file to clarify the rule
4. **Provide feedback**: Tell AI "That violates OUTPUT_RULES.md ¬ß2.1..."

### AI Seems Confused
1. **Ask clarifying questions**: "Are you referencing /.ai/SYSTEM.md?"
2. **Provide context**: "We use design tokens from _tokens.scss"
3. **Cite specific rules**: "Per OUTPUT_RULES.md ¬ß2.2, use var(--color-primary)"

### Instructions Conflict
1. **Hierarchy applies**: SYSTEM.md > COMPLIANCE.md > other domain files
2. **Legal overrides preferences**: COMPLIANCE.md beats STYLE.md
3. **Report conflict**: File an issue or update SYSTEM.md to clarify

### Need to Add New Rule
1. **Determine domain**: Behavioral? Technical? Legal? Code quality?
2. **Edit appropriate file**: SYSTEM / DOMAIN / STYLE / COMPLIANCE / OUTPUT_RULES
3. **Use clear sections**: Add under existing heading or create new ¬ß
4. **Provide examples**: Show correct/incorrect code or copy
5. **Commit with message**: `docs(ai): add rule for [specific case]`
